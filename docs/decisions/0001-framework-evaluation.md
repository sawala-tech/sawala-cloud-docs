# Sawala Cloud Public Docs — Framework Evaluation

Architecture Decision Record (ADR) for picking the framework that will power `https://docs.sawala.cloud/`.

This file is the working artefact of [`PLAN-public-docs-framework.md`](https://github.com/sawala-tech/sawala-cloud/blob/main/docs/plan/PLAN-public-docs-framework.md) Milestones 1 (this file), 2 (Candidates section), 3 (Docusaurus scoring), and 4 (Fumadocs scoring + decision). It gets committed into the new `sawala-cloud-docs` repo at `docs/decisions/0001-framework-evaluation.md` in M5 and stays there as the historical record of *why* we picked what we picked.

**Status at file creation (2026-05-16):** Requirements frozen. Candidates section pending M2. Scores pending M3/M4. Decision pending M4.

## Goal

Pick the framework that maximises (a) reader experience for Sawala customers, integrators, and partners reading docs, and (b) contributor velocity for the Sawala team and OSS contributors writing those docs — measured by the 14 requirements below.

The framework must serve four content surfaces ([Diátaxis](https://diataxis.fr/) shape): Tutorials, How-to Guides, Reference (including OpenAPI), and Explanation. It must be deployable on Cloudflare. It must look and feel like part of the `sawala.cloud` product family from day one.

## Requirements

Fourteen requirements. Each has a weight (`must` = 3×, `should` = 2×, `nice` = 1×) and a plain-language definition of the observable property. The scoring rubric is `0 (fails)` / `1 (works with effort)` / `2 (works out of the box)`. A candidate's final score is the weighted sum.

Weighted maximum = (8 musts × 2 × 3) + (3 shoulds × 2 × 2) + (3 nices × 2 × 1) = **66**.

### Must (8 requirements, weight 3×)

**R1 — Multi-language content.** Same page in Indonesian at `/{slug}` and English at `/en/{slug}`, with a header language switcher. Defined by behaviour: opening `docs.sawala.cloud/kontena/tutorial-blog` shows Indonesian; clicking the EN switcher navigates to `docs.sawala.cloud/en/kontena/tutorial-blog` and shows English. Matches `umahramah-next`'s pattern.

**R2 — Versioning.** Each product section supports a version selector (e.g. Kontena `v0.1`, `v0.2`). Selecting an older version serves the historical content of that version, not the latest. Defined by: a customer pinned to `@sawala/kontena-client@^0.1.0` can read the docs matching their installed version even after `v0.2` ships.

**R3 — MDX with custom components.** Pages can use `<Callout>`, `<Tabs>`, `<Steps>`, and our own `<ApiKeyPicker />`. The framework must let us import a React (or framework-equivalent) component from a sibling file and use it inline in MDX.

**R4 — Full-text search.** Header search box opens an instant-results overlay. Typing "API key" returns every page mentioning API keys, ranked by relevance. Built-in or via Algolia DocSearch (free for OSS docs).

**R5 — Cloudflare deploy.** Site deploys to Cloudflare Pages or Workers via a single `wrangler deploy` or Pages git integration. Verified by: pushing to `main` produces a public-URL deploy on the Sawala Cloudflare account.

**R6 — Code blocks with copy + multi-language tabs.** A `curl`, a `fetch`, and a `@sawala/kontena-client` example for the same operation render as three tabs in one block. Every code block has a "Copy" button.

**R7 — OpenAPI rendering.** We can drop a `<Scalar spec="…/kontena.openapi.json" />` (or framework-equivalent) on a page and it renders a reference comparable to the dashboard's `/dashboard/kontena/api-docs` Scalar embed. The framework's own OpenAPI renderer is acceptable if quality is comparable.

**R8 — Sidebar generated from frontmatter.** Adding `kontena/how-to-mint-api-key.mdx` with `title:` and `order:` frontmatter automatically slots it into the Kontena sidebar at the right place, without editing a central nav file.

### Should (3 requirements, weight 2×)

**R9 — "Edit this page on GitHub" link.** Every page footer has a link that opens the source MDX file in GitHub's editor on a new branch. Lowers contribution friction.

**R10 — Sawala brand fit.** Tailwind v4 with Sawala tokens works out of the box. Header/sidebar look like part of the `sawala.cloud` family — Sawala wordmark, brand gradient, Inter/Spectral type pairing.

**R11 — Dark mode.** Theme toggle persists per-user. Matches the dashboard.

### Nice (3 requirements, weight 1×)

**R12 — SEO defaults.** Autogenerated sitemap, sensible robots, default OG image with page title overlay.

**R13 — TypeScript-first.** Config files and any plugin authoring is TypeScript, not JavaScript.

**R14 — Active maintenance.** A release in the last 90 days *or* 5+ contributors active in the last 6 months.

## Scoring Rubric

| Score | Meaning |
| --- | --- |
| 2 | Works out of the box — first-party feature or canonical plugin, ~zero glue code |
| 1 | Works with effort — possible but requires custom code, plugin authoring, or a non-canonical workaround |
| 0 | Fails — the requirement cannot be met without forking the framework or building a parallel pipeline |

Weighted score = Σ (raw_score × weight_multiplier), where weight_multiplier is 3 for `must`, 2 for `should`, 1 for `nice`.

## Tie-Break Rules (in order)

1. Total team familiarity with the framework's stack (Sawala is a TypeScript/React/Next.js shop; a Next.js-native framework is a known quantity).
2. Ease of swapping out later if we regret the choice (MDX content portability is the key dimension — a framework that mangles MDX with proprietary directives is harder to leave).
3. Sawala brand fit assessed by the team after looking at the deployed prototype.

## Candidates

Seven realistic candidates, surveyed 2026-05-16. Evidence comes from each project's homepage, getting-started docs, and GitHub repo (release dates, license, maintenance signals). Versions cited are the latest at survey time.

### 1. Docusaurus (`@docusaurus/core`)

| Property | Value |
| --- | --- |
| Version | 3.10.1 (released 2026-04-30) |
| Stack | React 18 + MDX + Webpack/SSG; pure static output |
| i18n | **Built-in** — `i18n: { defaultLocale, locales: [...] }` in config; per-locale folders under `i18n/{locale}/` |
| Versioning | **Built-in** — `docusaurus docs:version 0.1` snapshots `docs/` into `versioned_docs/version-0.1/`; version switcher in UI |
| Search | Algolia DocSearch (free for OSS; application + approval cycle) |
| MDX | First-class; ships with `<Tabs>`, `<TabItem>`, admonitions; React components importable inline |
| Tailwind | Not native; supported via PostCSS plugin config but theme is React + CSS modules by default |
| Cloudflare deploy | Pure static → `wrangler pages deploy build` works directly |
| License | MIT |
| Maintenance | 64.9k stars, release 17 days ago, 156 releases total, 6,612 commits — robust |

Verdict: **Advance to prototype.** User asked about this one. Strongest built-in versioning of any candidate; the i18n + versioning combo is its core differentiator.

### 2. Fumadocs (`fumadocs-ui` + `fumadocs-mdx`)

| Property | Value |
| --- | --- |
| Version | Multi-package monorepo; `fumadocs-ui` and `fumadocs-mdx` are the relevant ones (sub-1.0 era still, sibling packages like `@fumadocs/sanity@0.0.4` released 2026-05-15) |

| i18n | **Built-in via `fumadocs-core/i18n`** with `defineI18n`, `createI18nMiddleware`, `defineI18nUI`. Per-file locale suffix (`page.fr.md`). Explicitly says it "manages only its own components" — `next-intl` can sit alongside |
| Versioning | **Not built-in.** Convention-based: separate routes per version (`/v0.1/...` vs `/v0.2/...`). No `docs:version` command. Cost is real for our R2 |
| Search | **Built-in via Orama** — runtime-bundled, client-side index |
| MDX | First-class via `fumadocs-mdx`; ships `<Callout>`, `<Tabs>`, `<Steps>`, `<TypeTable>`, `<File>` |

| Cloudflare deploy | Via `@opennextjs/cloudflare` (same path `sawala-cloud-ui` already uses) |
| License | MIT |
| Maintenance | Active — releases within the last 24h at survey time; 79.1% TypeScript |

Verdict: **Advance to prototype.** Closest stack-fit (same Next.js 16 + OpenNext + Tailwind v4 the rest of Sawala runs on). Versioning gap is the headline risk to validate in M4.

Verdict: **Advance to prototype only if Docusaurus and Fumadocs scores are within 3 points** (per M4's tie-break clause). Different stack (Astro, not React), so a real prototype is the only way to honestly score brand-fit and team-friction.

## Shortlist

Three candidates advance to prototype phase:

1. **Docusaurus** — answer the user's explicit question with evidence; strongest built-in versioning; track record at this surface.
2. **Fumadocs** — Next.js-native, Tailwind v4-native, stack-fit gives it an inherent contribution-velocity edge; versioning gap is the risk to validate.
3. **Astro Starlight** — *conditional* (built in M4 only if Docusaurus and Fumadocs scores are within 3 points); different stack so worth measuring honestly only if the React decision is genuinely close.

Ruled out: Mintlify (SaaS / OSS posture), MkDocs Material (Python stack), VitePress (Vue stack), Nextra (Fumadocs covers same niche better).

Weighted math: musts (R1-R8) = (2+2+2+1+2+2+1+2)×3 = 14×3 = **42**; shoulds (R9-R11) = (2+1+2)×2 = 5×2 = **10**; nices (R12-R14) = (2+2+2)×1 = **6**. Total = 42+10+6 = **58**. Correction to table: **58 / 66**.

What was painless:

- The four-quadrant content layout in `docs/kontena/{what-is-kontena, tutorial-first-blog, how-to-mint-api-key, reference/api}.mdx` — frontmatter-driven sidebar generation meant zero sidebar config to add a page.
- `npx docusaurus docs:version 0.1` *just worked* — versioned both `docs/` and `i18n/en/...`. This is Docusaurus' core differentiator and it earns it.
- Cloudflare Pages deploy: 3 commands (`project create`, `pages deploy build`, `curl -I` smoke). No build adapter, no SSR concern, no `wrangler.toml`.
- MDX components — `<Tabs>` for the curl/fetch/SDK code samples, `:::info`/`:::warning` admonitions — shipped in the box.

What fought us:

- **Scalar embed**. The React component (`@scalar/api-reference-react`) doesn't play with Docusaurus' SSG cleanly — needed `<BrowserOnly>` wrapper + dynamic script injection + `dangerouslySetInnerHTML` for the web component. Workable but ugly. The Sawala-canonical answer is probably `docusaurus-plugin-openapi-docs` (a separate plugin that generates MDX from OpenAPI), which trades the Scalar look for native Docusaurus theming — different visual identity from the in-dashboard reference at `/dashboard/kontena/api-docs`.
- **Theme is not Tailwind**. Docusaurus theming is React + CSS modules + Infima CSS variables. Sawala's brand stack is Tailwind v4 + shadcn. Brand alignment is achievable (we did it with ~30 lines of CSS), but reusing components from `@sawala/ui` would require Docusaurus theme swizzling — a meaningful one-off cost. A Tailwind-native framework would skip this.
- **Search is not turnkey**. Algolia DocSearch is the documented path but requires application + approval. The community plugin `@easyops-cn/docusaurus-search-local` is the fallback, but it's not first-party.

What was painless:

- **Tailwind v4 native** — the same Tailwind major sawala-cloud-ui uses. Brand integration would be a `@import` of Sawala's token CSS plus a Logo component swap.
- **MDX component vocabulary** is richer than Docusaurus' defaults: `<Steps>`, `<Tabs>` with named values, callout variants — all worked first try with no theme swizzling.
- **Search is turnkey** — `--search orama` at scaffold + zero application/approval cycle. Hot-reloads as you write content.
- **Scalar embed is a clean React import** in a `'use client'` component. No SSR workaround. Same component the dashboard already uses, so visual identity matches.
- **Built-in features Docusaurus doesn't have**: per-page OG image generation, `llms.txt`/`llms-full.txt`, markdown content endpoints (`/llms.mdx/.../content.md`) for AI agent consumption. These are *new* differentiators if AI/LLM accessibility of docs is a priority for Sawala.

What fought us:

- **Cloudflare deploy is a multi-step setup.** Three new config files plus deleting the scaffold's Node middleware (`proxy.ts`, which handles markdown content negotiation — a feature we lose by removing it). Once configured, deploys are `wrangler deploy`. The first deploy was four extra steps over Docusaurus' single `wrangler pages deploy`.
- **i18n is a real refactor**, not a config block. Moving the route tree under `[lang]/`, adding `createI18nMiddleware`, adding `defineI18nUI`, and naming files `*.en.mdx` is a structural change to the scaffolded app — multiple file edits. Doable, but the cost is real and visible.
- **Versioning has no first-party command.** The "copy directories + manual switcher" approach is workable but is the kind of bespoke convention that makes the project's own contributors learn an undocumented pattern.
- **Build errors are sometimes opaque.** Shiki rejected `env` as a language code without naming `how-to-mint-api-key.mdx:55` directly — required tracing the error message; the fix was `env` → `dotenv`. A novice contributor would not have caught this without help.

### Astro Starlight (tie-break only — built only if |Docusaurus − Fumadocs| ≤ 3)

**Skipped after analysis.** Docusaurus 58 − Fumadocs 55 = 3 (boundary case, the rule says "≤ 3"). The plan's tie-break clause triggers. However, all three plan-level tie-break rules (in order: team familiarity, swap-out ease, brand fit) point to Fumadocs unambiguously over Docusaurus *and* over Starlight (Astro is yet further from Sawala's React/Next.js daily stack than Docusaurus' React). A Starlight prototype would not change the decision; it would only add an asymmetric +1 to a third candidate that loses on every tie-break.

Decision to skip is recorded in this file's Decision section below and in the plan's Decision Log.

## Decision

**Winner: Fumadocs.**

**Runner-up: Docusaurus** (3 points higher on raw matrix, lost on tie-breaks).

**Score deltas that drove the call:**

- Docusaurus 58 / 66 vs Fumadocs 55 / 66 — Docusaurus wins the matrix by 3 points (boundary of the tie-break clause).
- The 3-point gap comes from R1 (i18n: Docusaurus 2 vs Fumadocs 1, +1×3 = +3) and R2 (versioning: Docusaurus 2 vs Fumadocs 1, +1×3 = +3) — totaling +6 for Docusaurus on the must-have surface.
- Fumadocs reclaims +3 from R7 (Scalar/OpenAPI: 2 vs 1, +1×3 = +3) and +1 each from R10 (brand fit: 2 vs 1, +1×2 = +2) on the should surface.
- Net Docusaurus advantage = 6 − 3 − 2 = +1 must surface + 0 in nices = 3 weighted points after both adjustments.

**Tie-break analysis (plan's rules, in order):**

1. **Team familiarity** — Fumadocs runs on Next.js 16 + React 19 + Tailwind v4, the *exact* stack of `sawala-cloud-ui`. Docusaurus is React 19 with Infima CSS (a stack Sawala does not use elsewhere). **Fumadocs wins by a wide margin.**
2. **Swap-out ease** — Both produce MDX in a `docs/` or `content/docs/` directory. Fumadocs' content is plainer (just frontmatter + body, no `sidebar_position` numbering), more portable to a hypothetical successor. **Slight Fumadocs lead.**
3. **Sawala brand fit** — Fumadocs is Tailwind v4 native and could share components from `@sawala/ui`. Docusaurus required CSS variable overrides and would need theme swizzling to share components. **Fumadocs wins clearly.**

All three tie-breaks favor Fumadocs.

**Forward-looking calls baked into the choice:**

- The i18n gap (R1: 1 vs 2) is acknowledged. M5 absorbs the cost: the bootstrap will set up `[lang]/` routing, middleware, and the locale-suffix file pattern *once*. After that, every new page is i18n-aware by default.
- The versioning gap (R2: 1 vs 2) is acknowledged. M6 cuts v0.1 by directory copy + a custom `<VersionSwitcher>` in the layout. The convention is documented in the repo README.
- The OpenNext deploy gap (R5: 1 vs 2) is acknowledged but cheap to absorb — `sawala-cloud-ui` already uses the same OpenNext-on-Cloudflare path; Sawala's CI patterns are reusable.

**Cloudflare deploy path chosen:** **Cloudflare Workers via `@opennextjs/cloudflare`** (same pattern as `sawala-cloud-ui`). Pages was the alternative if a static-only candidate had won — for Fumadocs (which has dynamic routes for search, OG, llms.txt), Workers is the correct surface.
